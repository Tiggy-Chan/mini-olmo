# mini-OLMo 项目大纲与发展路线

> 目标：在单卡 RTX 4060（8GB）+ WSL 环境下，从零实现一个 OLMo 风格的语言模型预训练流水线，并在可承受的时间内把模型能力推到「明显有语言组织能力和一定常识」的水平。

---

## 1. 硬件条件与核心约束

- **GPU**：NVIDIA GeForce RTX 4060（8GB 显存）
- **环境**：WSL Ubuntu + Conda + PyTorch + CUDA
- **影响：**
  - 显存上限 → 限制了可训练的参数量、batch size、序列长度；
  - 单卡 → 不能简单做模型并行 / 数据并行扩展到数十亿参数，只能在单卡上挤压；
  - 个人时间预算 → 训练时间控制在「十几小时到一两天」级别更现实。

在这个前提下，我们设计了一个**渐进式 scaling 路线**，从 ~26M → ~55M → ~100M，并预留出向上做 v4 的空间。

---

## 2. 当前进度总览（v1 / v2 / v3）

- **v1：mini-OLMo v1（约 26M 参数）**
  - 配置：8 层、384 维、12 头，d_ff ≈ 1536。
  - 数据：Wikitext（`corpus_name=wikitext`）。
  - 作用：完全打通 pipeline（数据 → tokenizer → 模型 → 训练 → 生成），做 sanity check。

- **v2：mini-OLMo v2（约 55M 参数）**
  - 配置：12 层、512 维、8 头，d_ff ≈ 2048。
  - 数据：初期在 Wikitext 上跑长训（几十 k steps）。
  - 作用：在不改变硬件的前提下，测试中等规模模型（>50M）在 8GB 显存上的稳定性与收敛速度。

- **v3：mini-OLMo v3（约 100M 参数）**
  - 配置：16 层、640 维、10 头，d_ff ≈ 2560。
  - 数据：扩展语料 `corpus_v2`（Wikitext + 英文 Wikipedia 子集）。
  - 训练：
    - 典型配置：`batch_size=2, grad_accum_steps=12, seq_len=512`；
    - 单步有效 token 数：≈ 2 × 12 × 512 ≈ 1.2×10^4；
    - 8 万步长训 ≈ 10^9 训练 token 量级，时间约 1 天左右。
  - 能力特征：可以稳定生成风格接近 Wikipedia 的英文段落（“百科式胡话”），语言组织能力明显高于 v1/v2。

---

## 3. 路线图：阶段与版本规划

### 3.1 阶段 0：环境与基础设施

- **目标**：在 WSL 上搭好完整的深度学习开发环境：CUDA、驱动、PyTorch、虚拟环境、基础工具链。
- **产物**：
  - `requirements.txt`
  - `src/mini_olmo/` 包结构
  - 基本 README / 文档目录

### 3.2 阶段 1：最小可用流水线（v1）

- **目标**：实现一个可端到端运行的小模型，验证所有环节打通。
- **子任务：**
  - 数据准备：`scripts/prepare_wikitext.py` → `data/raw/wikitext/*.txt`；
  - tokenizer：`tokenizer/train_tokenizer.py`（32k BPE）→ `tokenizer/tokenizer.json`；
  - 模型定义：`MiniOlmoConfig` + `MiniOlmoModel`（v1 配置）；
  - 数据管道：`LMDataset` / `create_dataloader`，seq_len=512，缓存 token ids；
  - 训练脚本：`scripts/pretrain.py`（AMP + AdamW + 余弦退火 + 梯度累积）；
  - 生成脚本：`scripts/generate.py`（从 checkpoint 加载并采样）。
- **状态**：已完成，并用于最初的 sanity check 训练。

### 3.3 阶段 2：规模提升与稳定长训（v2）

- **目标**：在不改变硬件的前提下，把模型做大到 ~50M，验证更大模型在 8GB 显存上的行为。
- **设计：**
  - v2 配置：12 层、512 维、8 头、d_ff=2048；
  - 仍然使用 Wikitext 作为训练语料；
  - 通过调节 `batch_size` 和 `grad_accum_steps`，在 8GB 显存内实现几十 k steps 级别的长训。
- **状态**：已完成配置与训练实验。

### 3.4 阶段 3：扩充数据 & 100M 模型（v3）

- **目标**：在 8GB 显存的限制下，推动到 ~100M 参数规模，并显著扩充训练数据，提高语言覆盖面。
- **关键改动：**
  - 数据：
    - 新脚本 `scripts/prepare_corpus_v2.py` → 构建 `corpus_v2`：
      - Wikitext 原始文本；
      - `wikimedia/wikipedia` 英文 20231101 子集；
      - 如果可用再加 BookCorpusOpen 子集（目前因 datasets 变更而可选）。
    - 所有语料统一写入 `data/raw/corpus_v2/{train,validation,test}.txt`。
  - 数据管道：
    - `LMDataset` / `create_dataloader` 支持 `corpus_name` 参数；
    - 首次使用某个语料时自动编码缓存到 `data/tokenized/{corpus_name}/*.pt`。
  - 模型：在 `pretrain.py` 中增加 `--model-size v3` 入口，约 100M 参数。
  - 训练：设计稳定不 OOM 的长训配置，控制在约 1 天训练时间。
- **状态**：v3 + corpus_v2 训练和生成已跑通。

### 3.5 阶段 4（可选）：向 v4 进一步 scaling

> 这一阶段是 “在你当前硬件上还能再往上挤多少” 的探索区间。

- **方向 1：再扩模型参数（v4）**
  - 目标参数量：**200M–300M 级别**，例如：
    - 24 层、768 维、12 头、d_ff≈3072；或
    - 18 层、896 维、14 头、d_ff≈3584。
  - 代价：
    - 需要更 aggressive 的显存技巧：gradient checkpointing、8-bit optimizer、减少 seq_len 等；
    - 训练速度会比 v3 明显慢（可能 2× 以上）。

- **方向 2：在 v3 上做更长 / 更聪明的训练**
  - 增加总训练 token：
    - 例如从 80k 步 → 160k 步，token 从 10^9 → 2×10^9；
  - 加入更丰富的数据：GitHub 代码、对话数据、简单 QA 语料等；
  - 做一些 task-oriented 的微调（翻译、摘要、问答）。

在 8GB 单卡场景下，**更现实、更高性价比的路线通常是“固定 v3 规模，继续加数据和训练时间”，而不是一味推到 300M+。**

## 4. 在当前硬件上的「极限模型规模」估计

这里给出一个基于经验的估算，重点是「在你这张 8GB 4060 上，模型规模 / 序列长度 / batch 的大致上限」。

### 4.1 已经验证可行的点

- **v3：~100M 参数**
  - 配置：16×640，seq_len=512；
  - 训练配置示例：
    - `batch_size=2, grad_accum_steps=12`（effective batch ≈ 12288 token/step）；
    - 使用 AMP + 梯度累积；
    - 显存占用约 6–7GB 左右（取决于具体配置和系统开销）。
  - 结论：**100M 规模在你这块卡上是稳定可用的 sweet spot**。

### 4.2 理论上还能挤到哪？

- **参数量方面：**
  - 粗略经验：单卡 8GB 上，带 AMP + 合理 batch/seq 时，**200M 级别**的解码器模型是可以尝试的，但：
    - 需要牺牲 batch / seq_len；
    - 必须开启 gradient checkpointing 等技巧；
    - 训练速度会非常慢，不适合作为「经常重来」的实验。

- **序列长度方面：**
  - 当前使用 `max_seq_len=512`；
  - 如果想在 v3/v4 上跑更长序列（比如 1024），需要进一步减小 batch/grad_accum 或使用更强的显存优化。

综合来看：

- **在你当前硬件（8GB 4060）上，比较“舒服”的上限是：**
  - 参数量：**100M 左右（v3 级别）**；
  - 在做一点工程优化后，可以实验性地摸到 **200M 左右**，但不适合作为主力工作马。

## 5. 在「没有硬件限制」条件下，我能帮你做到什么规模？

这里的前提是：

- 你有足够的算力（多卡 A100 / H100 或云算力），可以承受数周训练；
- 你愿意按照工程实践把分布式训练、数据流水线、监控等都搭起来；
- 我负责：
  - 模型 / 数据 / 训练策略的设计；
  - 代码结构与实现建议；
  - 调参与调试思路；
  - 监控指标与 early stopping / checkpoint 策略；
  - 任务和评估设计。

在这种前提下：

- **参数规模方面：**
  - 我可以帮助你设计和实现从 **几亿参数到数十亿参数** 的 OLMo/GPT 风格模型：
    - 例如 1B、3B、7B 这类典型档位；
    - 再往上（10B–70B）主要受工程复杂度和资源成本限制，而不是「我能不能写/指导这类代码」。

- **训练与数据方面：**
  - 规划多 TB 级别的语料清洗和混合（类似 Dolma）；
  - 设计分阶段训练（小序列 → 长序列、多阶段 LR schedule）；
  - 建立多任务评测（语言建模困惑度、多种下游任务、对话评测）。

**总结一句：在没有硬件限制时，我在算法与工程指导上的「上限」，基本可以覆盖到你想模仿的公开大模型（比如 OLMo、LLaMA 7B/13B 量级）的完整训练流程。真正的瓶颈会变成：算力 / 时间 / 数据许可，而不是“我不知道怎么做”。**

## 6. 项目未来发展脉络（建议版本线）

### 6.1 短期（当前周期）

- **巩固 v3：**
  - 跑完 80k 步或更多（如 120k / 160k）；
  - 用多样 prompt 系统性地感受生成质量；
  - 记录不同 checkpoint（60k/80k/100k）的 loss 和主观体验。

- **数据分析：**
  - 粗算 `corpus_v2` 的 token 量级，理解「每次长训走了多少数据」；
  - 决定是否需要进一步扩充语料（更多维基、其他开放语料）。

### 6.2 中期：微调与任务化

- **下游任务微调：**
  - 在简单任务上微调 v3：
    - 英文分类（IMDb / SST-2）；
    - 简单 QA / Cloze 风格任务。

- **对话/指令微调（轻量版）：**
  - 收集一小批指令-回答样本；
  - 在 v3 上做 LoRA / Adapter 微调，体验「聊几句」的能力。

### 6.3 中长期：v4 & 更大规模探索

- **v4（200M 级别）的 feasibility study：**
  - 先做纯理论估算 + 显存 profile；
  - 如果可行，搭 gradient checkpointing / 8-bit optimizer；
  - 尝试小步近似预训（比如 10k 步）评估收益与成本。

- **更广泛的数据混合：**
  - 加入代码数据（GitHub 子集）、多语种数据；
  - 探索简单的对齐与过滤策略（去重、质量打分）。
