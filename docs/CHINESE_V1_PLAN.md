# mini-OLMo 中文优先 V1 开发方案

## 1. 目标定义

本版本不追求“智能涌现”，目标收敛为：

- 在单张 `RTX 4060 Laptop 8GB` 上完成一套可复现的中文优先训练流程。
- 训练出一个以中文为主的小参数语言模型，具备基本的中文续写、问答、角色跟随能力。
- 在简单单轮对话中，输出比当前英文百科续写模型更稳定、更像“助手”而不是“乱续写”。

本版本的目标上限：

- 中文能力显著强于其他语言。
- 可以保留少量英文、数字、URL、代码片段能力，但不追求多语言平衡。
- 多轮复杂推理、长上下文工具调用、对齐稳定性，不作为 V1 交付目标。

## 2. 当前项目现状

当前仓库已经具备：

- 基础 decoder-only Transformer 结构。
- 原始语料准备、tokenizer 训练、预训练、生成脚本。
- 适用于英文 Wikipedia / Wikitext 风格语料的最小可用训练流水线。

当前仓库的主要问题：

- 语料以英文为主，不满足中文优先目标。
- tokenizer 仅从 `wikitext` 训练，且当前方案更偏向英文空格分词场景。
- 训练目标是纯续写，不包含中文指令微调，因此模型不会自然具备“助手式对话能力”。
- README 中 checkpoint 描述与仓库内实际权重配置存在不一致，后续需要顺手清理。

## 3. V1 总体策略

V1 采用“两阶段训练”路线：

1. 中文预训练
   先训练一个中文优先的 base model，让模型掌握中文词法、句法、常识表达和常见书面风格。
2. 中文指令微调
   再用高质量中文问答/对话数据微调，让模型学会按“用户/助手”格式回答。

核心判断：

- 只做预训练，模型最多像“中文续写器”，不会自然成为“中文聊天助手”。
- 对 `26M-100M` 级别模型来说，中文单语或近单语路线，比中英混训更有性价比。

## 4. 版本规划

### 4.1 快速验证版：`v2-cn`

用于验证中文语料、tokenizer、训练稳定性和基础生成质量。

- 结构：沿用现有 `v2` 档位思路。
- 目标规模：约 `50M-60M`。
- 作用：
  - 快速打通中文版数据流水线。
  - 用较低成本检查 tokenizer、损失曲线和中文生成是否正常。
  - 作为后续 `v3-cn` 的风险前置验证。

### 4.2 主力版本：`v3-cn`

作为 V1 主要交付版本。

- 结构：沿用现有 `v3` 档位思路。
- 目标规模：约 `90M-110M`。
- 作用：
  - 在 8GB 显存约束下尽量逼近可用中文小模型的上限。
  - 为后续中文 SFT 提供更好的底座。

## 5. 数据策略

### 5.1 语料原则

- 中文占比目标：`95%+`。
- 保留少量英文与符号文本，避免模型完全失去基础英文术语和链接格式能力。
- 先以高质量、开放来源、结构清晰的数据为主，不急着上大而杂的抓取网页。
- 先保证干净，再谈规模。

### 5.2 V1 预训练语料配方

第一版预训练语料建议采用以下配方：

- `40%` 中文维基百科
- `20%` 中文维基文库
- `30%` 高质量中文网页/教育类语料抽样
- `10%` 中文问答、教程、科普、技术类补充文本

纯顶级口径下的主训练集进一步收紧为：

- 中文维基百科
- `FineWeb Edu Chinese 4_5`
- 你自己人工筛选过的中文精选文本

优先来源：

- 中文维基百科 dump
- 中文维基文库 dump
- 中文高质量开放网页语料抽样
- 可明确确认来源和许可的中文知识文本

暂不建议：

- 一上来就吞超大规模混杂中文网页全量数据
- 来源不清或难以确认用途边界的数据直接混入主训练集
- 为了凑体量引入太多英文或多语文本

### 5.3 预训练集大小目标

第一版不追求 TB 级数据，先追求“4060 上可反复实验”：

- 第一阶段目标：清洗后 `5GB-15GB` 纯文本
- 第二阶段扩展：清洗后 `20GB-50GB` 纯文本

说明：

- 对当前硬件而言，高质量 `5GB-15GB` 中文语料通常比低质量几十 GB 更有价值。
- 在 tokenizer 合理的情况下，这已经足够支撑小模型做出明显的中文风格迁移。

当前执行口径：

- 直接构建 `zh_corpus_v3_elite_20gb`
- 目标体量先卡在 `20GiB`，避免一开始把磁盘和时间成本拉得过高
- 主训练集只使用 `Wikipedia + FineWeb Edu Chinese 4_5`
- 不再默认混入 `Cosmopedia`
- 不再默认混入 `3_4 / 2_3` 这类低一级分桶
- 如果要继续提升质量，优先追加人工精选中文文本，而不是放宽数据桶位
- 大语料阶段默认使用磁盘去重，而不是内存去重
- 当前大语料构建脚本还不支持断点续跑，提交前需要明确写进文档

### 5.4 第二阶段聊天数据配方

阶段二不再追求“像预训练那样尽可能大”，而是追求高质量、格式统一、中文占比高。

V1 默认聊天数据配方：

- `BelleGroup/train_1M_CN` 抽样
- `FreedomIntelligence/alpaca-gpt4-chinese` 抽样
- 仓库内置少量中文种子样本
- 用户自己补充的高质量本地 `json/jsonl` 对话数据

阶段二数据清洗原则：

- 统一到 `messages` 结构
- 过滤中文占比过低的样本
- 过滤过短、过长、轮数异常和重复样本
- 默认过滤“作为一个 AI 语言模型”这类低质量 boilerplate
- 保留单轮和短多轮对话，先服务于“回答清楚”的目标

## 6. tokenizer 策略

当前 tokenizer 路线不适合直接复用于中文版本，V1 需要单独重做。

目标：

- 明显提升中文切分质量。
- 保证中英混合、数字、标点、URL 的基本可处理性。
- 降低 embedding 参数浪费。

方案：

- 新增中文 tokenizer 训练脚本，不复用当前仅面向 `wikitext` 的训练入口。
- 优先采用适合中文连续文本的方案。
- 词表大小优先尝试 `16k`，必要时对比 `24k`。
- tokenizer 训练语料直接来自中文预训练语料，而不是英文 `wikitext`。

V1 选择原则：

- 如果工程复杂度要尽量低，先做一版稳定中文 BPE。
- 如果允许增加依赖，则优先考虑更适合中文的 tokenizer 方案。

## 7. 训练策略

### 7.1 训练阶段拆分

#### 阶段 A：中文数据与 tokenizer 验证

- 准备中文原始语料
- 清洗、去重、切分 train/validation
- 训练中文 tokenizer
- 做少量编码采样检查

验收标准：

- 中文文本不会被异常切碎
- 标点、数字、英文缩写、URL 能被合理编码
- 常见中文 prompt 的 token 长度可接受

#### 阶段 B：中文 base model 快速预训练

- 先训练 `v2-cn`
- 先跑 smoke test，再跑中等步数训练
- 重点检查 loss 是否正常下降，中文生成是否已经明显优于当前英文模型迁移到中文的表现

验收标准：

- loss 曲线稳定
- 中文续写可读
- 不会大量退化为乱码、标点循环、英文乱入

#### 阶段 C：中文主力预训练

- 用 `v3-cn` 跑主训练
- 以 8GB 显存可承受的 batch、梯度累积和序列长度为准
- 先确保能完整跑通并保存 checkpoint，再逐步拉长训练

验收标准：

- 可以稳定输出中文短段落
- 能围绕中文 prompt 做基本续写
- 中文占主导时，不再频繁漂移到英文百科风格

#### 阶段 D：中文指令微调

- 引入中文问答、助手回复、角色格式样本
- 将训练样本统一为 `user/assistant` 或等价 chat 模板
- 训练一个单独的 SFT checkpoint

当前默认实现：

- 使用 `prepare_sft_zh.py --recipe chat_v1` 构建阶段二数据
- 先做中文短对话和说明类回答
- 不急着把复杂工具调用、多轮代理轨迹混进 V1

验收标准：

- 对“你好，请介绍一下你自己”“什么是机器学习”这类问题能直接按助手口吻回答
- 回答长度可控
- 跑偏到百科段落的概率明显下降

## 8. 模型与训练配置建议

### 8.1 `v2-cn` 建议

- 用于快速验证
- 序列长度维持 `512`
- 先保证能高频迭代，不追求第一步就上满模型规模

### 8.2 `v3-cn` 建议

- 作为主要交付
- 序列长度先维持 `512`
- 词表缩小后，embedding 参数会下降，有利于把预算更多留给主体网络

### 8.3 工程优先级

先做：

- 中文 tokenizer
- 中文语料准备
- `v2-cn` / `v3-cn` 入口
- 中文生成与评测脚本

后做：

- gradient checkpointing
- 更长上下文
- 更复杂采样策略
- 更大模型档位

## 10. 数据格式规划

### 10.1 预训练数据

统一落地为纯文本或 `jsonl` 文本字段，核心要求是简单稳定，便于流式读取。

建议字段：

```json
{"text": "这里是一段清洗后的中文文本。"}
```

### 10.2 SFT 数据

统一为明确的对话结构，不再混入百科式纯文本。

建议字段：

```json
{
  "messages": [
    {"role": "user", "content": "你好，请介绍一下你自己。"},
    {"role": "assistant", "content": "我是一个中文小模型，目前擅长基础中文问答和续写。"}
  ]
}
```

## 11. 评测方案

V1 不做复杂 benchmark，先做“可用性评测”。

### 11.1 预训练阶段评测

- validation loss
- 固定中文 prompt 生成样例
- 常见中文知识、定义、解释类 prompt 的主观观察

### 11.2 SFT 阶段评测

固定一组 `30-50` 条中文测试 prompt，覆盖：

- 自我介绍
- 常识问答
- 定义解释
- 步骤说明
- 摘要改写
- 简单代码解释
- 礼貌拒答或不确定性表达

验收时重点看：

- 是否按助手口吻回答
- 是否能围绕问题作答
- 是否明显减少英文漂移
- 是否减少无关续写

## 13. 风险与取舍

### 13.1 数据风险

- 中文网页语料噪声高
- 版权和使用边界需要控制
- 问答数据质量差时会直接污染回答风格

### 13.2 模型风险

- 小模型容量有限，不能把目标定成“全能聊天模型”
- 仅靠预训练无法解决聊天格式跟随问题
- 中文 tokenizer 方案选错，会明显拖累效果

### 13.3 工程风险

- 单卡 8GB 容易受 batch、seq_len、词表大小影响
- 数据规模一旦拉太大，实验周期会迅速失控
- 如果不先做 `v2-cn` 验证，直接上 `v3-cn` 会放大排错成本

## 14. 开工顺序

正式进入实现时，严格按下面顺序执行：

1. 写中文数据准备脚本
2. 写中文 tokenizer 训练脚本
3. 扩展数据加载与训练入口
4. 跑 `v2-cn` smoke test
5. 跑 `v2-cn` 中等训练
6. 产出并检查中文生成样例
7. 跑 `v3-cn` 主训练
8. 写中文 SFT 数据准备与训练脚本
9. 产出第一版中文聊天模型

## 15. V1 完成交付标准

满足以下条件即可认为第一版完成：

- 仓库内存在完整中文训练文档和脚本入口
- 能从中文语料开始训练 tokenizer 和 base model
- 能产出至少一个中文 base checkpoint 和一个中文 SFT checkpoint
- 对常见中文问答 prompt，模型能输出基本清晰、基本相关的中文回答
- README 已更新到与实际能力、配置、权重一致
